### ~~C1. `PUT /owned-shoes/{id}` can arbitrarily overwrite the mileage ledger~~
**Struck 2026-07-07 (Phase 2 Session C — `r2:` commit; see `docs/changelog.md`).** The mileage-ledger breach is closed: `current_mileage` and `starting_mileage` are removed from `OwnedShoeUpdate`, so the blind-setattr PUT can no longer touch the ledger. Manual corrections now go through the sanctioned `rotation.adjust_mileage()` behind `POST /owned-shoes/{id}/adjust-mileage`, which records the override as a journal note (`triggered_by="mileage_adjustment"`) so reconciliation can explain the drift — the third blessed exception per domain_model §4.5. Both UI call sites (the ShoeDetail adjust dialog, the OwnedShoeForm edit field) were repointed/removed. Regression: `tests/test_owned_shoes.py`. **Residual (not this fix):** the `status` pattern-validation rider folds into M2 (vocabulary validation), and the `starting_mileage` delta-application concern is moot now that it's immutable post-create (the UI already disabled it on edit). Original finding preserved below.

**Description.** `OwnedShoeUpdate` (schemas.py) declares `current_mileage: Optional[float]` as a writable field, and `update_owned_shoe` (routers/owned_shoes.py) applies `model_dump(exclude_unset=True)` with a blind `setattr` loop. Any PUT can set `current_mileage` to any value. `starting_mileage` is equally writable, and editing it does **not** adjust `current_mileage`, so even a well-intentioned correction breaks the identity `current_mileage = starting_mileage + Σ(attributed distances)`.
**Why it matters.** This is a direct violation of domain_model.md §4.5/§4.6 — the mileage ledger with its single sanctioned write path is called "the domain's strongest integrity guarantee," and Phase 5 shipped under an explicit "counters untouched" promise. One generic update endpoint quietly provides the parallel write path that B7 exists to forbid. It also silently invalidates cost/km, the retirement pipeline, checkpoint detection, and every displayed total. Per the review's ground rules, an invariant violation is Critical by definition; the mitigations are that the app is single-user and the current UI *probably* doesn't send the field — but the API contract, MCP-adjacent tooling, and any future client all can.
**Suggested solution.** Remove `current_mileage` from `OwnedShoeUpdate`. Route `starting_mileage` edits through a small `rotation` function that applies the delta to `current_mileage` in the same transaction (preserving the identity). While in the schema: `status` has no value validation on create or update (compare `PlannedRaceUpdate`, which patterns its status) — add `pattern="^(active|retired|for_sale)$"` so a typo can't silently drop a shoe out of the pipeline/feeds. **Verified during the Prompt-7 frontend pass (2026-07-05): the UI sends this field deliberately.** `ShoeDetail.jsx` has an "Adjust mileage" dialog (two-step confirmation, warning that an override "creates a discrepancy the next reconciliation will surface") and `OwnedShoeForm` exposes an editable Current-mileage field on edit. That reframes this from "open hole" to **undocumented invariant exception** — domain_model.md §4.5/§4.6 admit no manual-override provision, so the constitution and the code disagree. Revised fix: keep the capability but make it sanctioned — a dedicated `rotation.adjust_mileage()` behind its own explicit endpoint (recording the adjustment so reconciliation can explain the drift), remove `current_mileage` from the generic blind-setattr update path (which accepts the field from any client with no confirmation — the two-step confirm lives only in the browser), repoint the two UI call sites, and record the escape hatch in domain_model.md §4.5 as a third blessed exception. The `starting_mileage` delta-application fix stands as written.
**Estimated effort:** Low. **Estimated risk:** Low (removing a field from an update schema; one focused UI check).

### C2. No auth on three mutation surfaces, `0.0.0.0` default bind — ✅ RESOLVED (R2.1, 2026-07-07)
**Resolution.** The security pass shipped (design_decisions **E7** ← E1; changelog 2026-07-07 Session D). A shared bearer token (`ANTON_SECRET`) gates `/api/*` **and** `/mcp` via one pure-ASGI middleware; the default bind moved to `127.0.0.1`; the app fails fast if the secret is unset. All three consumers send it — the SPA (`VITE_ANTON_SECRET`, axios interceptor + the raw chat `fetch`/scrape-SSE paths), Claude Desktop (`mcp-remote --header`), and the loopback self-connection (injected at connect time, scoped so the secret never leaks to an external MCP server — the silent-break risk this finding called out). `POST /api/chat/resource/read` (the arbitrary-URI proxy this finding sharpened) is covered like every other endpoint. 13 HTTP-layer tests in `tests/test_auth.py`.
**Intentionally deferred (not part of R2.1):** (a) **CORS tightening** — left at `allow_credentials=True` + `allow_methods/headers=["*"]`. `allow_origins` is already restricted to the two SPA origins, and the `allow_headers=["*"]` wildcard is precisely what lets the browser send the `Authorization` header cross-origin; narrowing it is pure hardening with real preflight-breakage risk and near-zero gain now that the token is the actual defense. Revisit only if the origin set changes. (b) **Rate limiting** on `/api/chat/message` — a separate R2 item (plan §6): R2.1 stops *anonymous* spend, not an authenticated client looping.
**Original description (kept as history).** `POST /api/chat/message` was an unauthenticated proxy that spent the server's paid LLM keys, `/mcp` accepted unauthenticated writes, every destructive REST endpoint was open, and `POST /api/chat/resource/read` was an unauthenticated arbitrary-URI proxy — all on a default bind that made "localhost-only" a property of the network, not the app.

### H2. Orphan retirement retires live deals on partial detail-fetch failure
**✅ RESOLVED (2026-07-08, Session O).** Fix (a) applied: `scrape_retailer_for_shoe` now tracks `searched_urls` (every URL the search returned) alongside `fetched_urls` (successful detail fetches) and orphan-retires against their **union** — so a product still listed in search keeps its live deal even when its detail fetch times out. The rename-cleanup behaviour is preserved (a URL that truly vanishes from search is in neither set). Covered by `test_orchestrator.py::test_partial_detail_failure_does_not_orphan_a_live_deal` (was the H1 xfail placeholder, now a real pass) plus the orphan-on-rename regression. B10's stated intent — "a transient scrape failure can never mass-extinguish deals" — is now fully delivered.

**Description.** In `orchestrator.scrape_retailer_for_shoe`, `seen_urls` accumulates only URLs whose `get_product_details` call **succeeded** (a `None` detail result `continue`s without recording the URL). `deactivate_orphaned_deals` then retires any active deal whose URL isn't in `seen_urls`. The non-empty guard (`if not seen_urls: return 0`) protects only the everything-failed case: if the search returns a product but its detail fetch times out (a 10-second `requests` timeout against slow retailer pages — routine), that product's live deal is retired this scrape.
**Why it matters.** B10's stated intent is that "a transient scrape failure can never mass-extinguish deals," and the interlock as written doesn't fully deliver it. The failure is partially self-healing (the next successful scrape re-creates the deal), but between scrapes the feed, Home top-deals, and replacement-deal counts silently under-report — and with manual-trigger scraping, "between scrapes" can be days. It also erodes `detected_at` honesty (the recreated deal looks new).
**Suggested solution.** Two candidate fixes; the first is safer. (a) Track a second set of *search-returned* URLs (canonicalized the same way details are — note search URLs and detail URLs already agree on the `/products/<handle>` shape in both Shopify and En Route scrapers) and orphan-retire against the union, so a product that appeared in search is never orphaned merely because its detail fetch failed. (b) Alternatively, skip the orphan pass whenever any per-product error was recorded this scrape — simpler, but delays legitimate rename cleanup. Add the H1 test for this exact case in the same change.
**Estimated effort:** Low. **Estimated risk:** Medium (this code path is the deals feed's self-cleaning mechanism; the rename-cleanup behavior it exists for must be preserved — hence tests first).

### H3. `DELETE /owned-shoes/{id}` bypasses rotation semantics; SQLite FK enforcement is off
**Description.** Deleting an owned shoe goes straight through ORM cascade (`OwnedShoe.runs` `cascade="all, delete-orphan"`), which deletes the attribution rows without ever touching `rotation.delete_run`. Consequences: (1) the underlying `coros`/`manual` activities are **not** deleted — they linger as unattributed rows in the unified training feed, which is inconsistent with the per-run delete path (where non-archive activities are removed) and with the shoe's disappearance; (2) `PlannedRace.planned_shoe_id` and `StravaGearMapping.owned_shoe_id` rows referencing the shoe become dangling — and because `database.py` never sets `PRAGMA foreign_keys=ON` (SQLite defaults it off, and SQLAlchemy doesn't enable it for you), SQLite accepts the dangling references silently. The relationship loads then quietly resolve to `None`, so nothing errors — the data is just wrong.
**Why it matters.** Domain_model.md §6 calls owned-shoe deletion "a destructive exception," but the code treats it as generic CRUD, and it's the one place the training domain's carefully asymmetric delete semantics (B11, §4.8) don't apply. The disabled FK pragma is broader than this endpoint: it means *every* FK in the schema is advisory, which quietly undercuts the integrity story the migration discipline works so hard for.
**Suggested solution.** (1) Add the standard `event.listens_for(Engine, "connect")` hook in `database.py` issuing `PRAGMA foreign_keys=ON` — one-time, and run a one-off `PRAGMA foreign_key_check` against the live DB first to surface any existing dangling rows before enforcement starts rejecting things. (2) Give shoe deletion a home in the service layer: `rotation.delete_owned_shoe(db, id)` that iterates attributions through the existing delete-run semantics (archive rows un-attribute, non-archive activities go), nulls `planned_shoe_id` / gear-mapping references, then deletes the shoe. The MCP surface has no delete-owned-shoe tool (correctly — 5.2 reserves this for the runner), so only the REST handler changes.
**Estimated effort:** Low–Medium. **Estimated risk:** Medium (destructive path + turning on FK enforcement over a live DB; do the `foreign_key_check` and add a test for the new service function).

### H4. The Phase-5 ShoeRun proxy N+1 is live in today's endpoints, not just a future-code trap
**Description.** The pattern is documented (dependency_graph.md §8.2, architecture.md §15.4) as a hazard for *new* code — but it is firing right now. `GET /owned-shoes/{id}/runs`, MCP `get_shoe_runs`, `draft_shoe_review`, and the runs/detail resources all `.join(Activity, ...)` **for ordering only**; a join does not populate the `activity` relationship, so when `ShoeRunResponse` / `_shoe_run_to_dict` read the property proxies, each row lazy-loads its Activity — one query per run. **Estimated effort:** Low. **Estimated risk:** Low (read-path only; identical results, fewer queries).

### ~~M3. `run_scrape_job` can wedge the scrape lock permanently~~
**Struck 2026-07-07 (Phase 2 Session C — `r2:` commit; see `docs/changelog.md`).** The whole body of `run_scrape_job` now runs under the lock-releasing `try/finally` (the setup block that used to sit before it is inside the try), so any failure after acquisition still releases the lock. `release_scrape_lock()` is now tolerant of an unheld lock (no more `RuntimeError` on double-release). Added the operational escape hatch `force_release_scrape_lock()` behind `POST /api/admin/scrape-lock/release` (`{"was_held": bool}`) and a synchronous `GET /api/scrape/status` (`{"scrape_running": bool}`). **Description.** In `scrape_runner.run_scrape_job`, the setup block (shoe/retailer queries, promo detection) executes **before** the `try/finally` that releases the scrape lock. The caller acquired the lock synchronously before scheduling the task; if setup raises (DB hiccup, anything), the function exits with the lock held. Every subsequent scrape — REST, MCP, background — gets 409/`started:false` until the process restarts, with nothing in the UI explaining why.
**Why it matters.** The lock's own docstring says its job is making failure "a clean, fast rejection instead" of a hang; this gap turns one transient setup error into a silent standing outage of the app's core data pipeline. Low probability, high confusion.
**Suggested solution.** Move `try:` above the setup block (or wrap the whole body) so the existing `finally: release_scrape_lock()` covers everything after acquisition. Consider making `release_scrape_lock` tolerant of an unheld lock (it currently propagates `RuntimeError` on double-release) while there.
**Estimated effort:** Low. **Estimated risk:** Low.

### M4. Per-shoe query fan-out on every rotation read
**Description.** `_attach_computed_fields` runs per shoe in `GET /owned-shoes` (and per shoe in COROS confirm responses); each call issues a `LIKE`-based image-match query plus a full activity load for lifetime stats — roughly 2–3 queries per shoe per list request. The rotation resource is also **pre-loaded into every chat conversation's system prompt**, so every Son of Anton conversation start pays the full fan-out.
**Why it matters.** Labeled personal-scale patterns are fine (CLAUDE.md §12), but this one isn't labeled, sits on the most-read aggregate in the app, and compounds with H4 on the same pages. **Suggested solution.** One grouped query (`GROUP BY shoe_run.owned_shoe_id` over the attribution→activity join returning avg pace-seconds, avg HR, count) feeding all shoes in a list at once; image matching either batched or cached per (brand, model) within the request. Do it inside the future shared owned-shoe shaping function (Task D's landing spot) so it's fixed once for REST + MCP + resource.
**Estimated effort:** Low–Medium. **Estimated risk:** Low (read-only; assert identical outputs against a snapshot).

### M5. The shipped `sync_coros_runs` prompt half-reflects the practiced protocol
**Description.** The prompt's Step 1 was updated to route around the dormant server-side path ("do NOT call `fetch_unsynced_coros_runs` — it requires API credentials that are not configured"), but Step 6 still instructs `confirm_coros_run`. The protocol actually practiced from Claude Desktop reportedly logs via `log_run_to_shoe`. If `confirm_coros_run` works (it needs no COROS credentials — it's a local write with dedup on `coros_activity_id`), the practiced protocol is leaving the dedup key behind: runs logged via `log_run_to_shoe` **without** `coros_activity_id` are invisible to the primary dedup tier and protected only by the date/±0.1 km fallback. If it doesn't work, Step 6 is a trap for any fresh agent following the prompt.
**Why it matters.** The prompt is versioned prose that *is* the sync feature (C6); a prompt that disagrees with practice in its write step is exactly the "third party renames a tool and nothing signals" fragility class §8.6 warns about, except self-inflicted. And the dedup asymmetry is a genuine never-double-count (§4.7) exposure.
**Suggested solution.** One short verification session: call `confirm_coros_run` once against a real synced run. If it works, fix the *practice* (and note that `log_run_to_shoe` could accept an optional `coros_activity_id` — `rotation.log_run` already takes one, the MCP tool just doesn't expose it, a two-line change that would make either path dedup-safe). If it doesn't, fix Step 6 and record why in the changelog. **Uncertainty note:** this review verified the code paths, not live tool behavior against Claude Desktop.
**Estimated effort:** Low. **Estimated risk:** Low.

### M6. `logging.basicConfig` at import time in `base_scraper.py`
**Description.** A library module calls `logging.basicConfig(level=logging.INFO)` at import. Because `main.py → mcp_server → scraper_manager → registry → subclasses → base_scraper` runs at boot (dep_graph §2), this line configures the **root logger for the entire application** as a side effect of importing a scraper base class.
**Why it matters.** It works today by accident of import order, silently owns app-wide log configuration from the least likely file, and will fight any future deliberate logging setup (an agents/observability session is on the roadmap). CLAUDE.md §8 describes module loggers as the convention; this is the one violation of it.
**Suggested solution.** Delete the line; put an explicit `logging.basicConfig` (or dictConfig) in `run.py` where process-level configuration belongs.
**Estimated effort:** Low. **Estimated risk:** Low (verify scraper INFO logs still appear when run via `run.py` and via pytest).

### L1. `trigger_scrape`'s advisory notification always reports zero deals
**Description.** In `mcp_server.trigger_scrape`, `results` is a **dict** (both `scrape_shoe` and `scrape_all_shoes` return dicts), but the summary code iterates it as a list: `sum(r.get("deals_found", 0) for r in results if isinstance(r, dict))` iterates the dict's *keys* (strings), so `deals_found` is always 0 and `len(results)` counts keys, not shoes.
